## Задание 1. Анализ работы Suricata при сетевом сканировании

### Цель
Проверить срабатывание IDS Suricata на типовые виды сетевого сканирования (Nmap) и попытки эксплуатации, оценить эффективность базовой конфигурации правил.

### Топология стенда
- Защищаемая VM (IDS): `10.0.2.5` (`ubuntu-m`), интерфейс `eth0`.
- VM злоумышленника: `10.0.2.19` (`ubuntu-n`).
- Подсеть: `10.0.2.0/24`.
- Среда: Yandex Cloud (Security Groups разрешают ICMP, TCP:22, весь трафик внутри подсети).

### Методика тестирования

Были выполнены следующие тесты:
1. Сканирование портов Nmap: `-sA`, `-sT`, `-sS`, `-sV` по IP `10.0.2.5`.
2. Попытка генерации сигнатурного события: HTTP‑запрос с полезной нагрузкой `/etc/passwd` на порт 80.
3. Проверка логов Suricata: `fast.log`, `eve.json`.
4. Проверка статуса службы и конфигурации интерфейса.

### Анализ работы Suricata. 
В ходе тестирования в логах Suricata зафиксированы только штатные статистические события (event_type: stats). Отсутствие алертов (alert: 0) обусловлено отсутствием загруженных правил детектирования (rules_loaded: 0).Fail2Ban не зафиксировал блокировок, поскольку в логах целевых сервисов (в частности, SSH) отсутствовали повторяющиеся ошибки аутентификации или иные триггерные события. Это подтверждает, что в рамках теста не происходило попыток брутфорса, на которые Fail2Ban мог бы среагировать. Для полноценного тестирования механизмов защиты необходимо: подключить и обновить правила для Suricata, а также смоделировать атаки (сканирование, подбор паролей), чтобы проверить срабатывание IDS и автоматическую блокировку через Fail2Ban.

### Анализ работы Fail2Ban. 
Проверка статуса jail sshd показала наличие 3 суммарных неудачных попыток аутентификации при нулевом количестве блокировок. Это соответствует ожидаемому поведению: Fail2Ban накапливает события в пределах окна findtime и применяет блокировку только при превышении порога maxretry. В данном случае порог не был достигнут, поэтому IP‑адреса не попали в бан.Такой результат подтверждает правильную настройку механизма защиты: система не реагирует ложноположительно на единичные или редкие ошибки, но готова заблокировать атаку при росте числа попыток. Для проверки срабатывания блокировки можно смоделировать дополнительные неудачные входы (например, через ssh с неверным паролем), чтобы превысить maxretry.
