## Задание 1. Анализ работы Suricata при сетевом сканировании

### Цель
Проверить срабатывание IDS Suricata на типовые виды сетевого сканирования (Nmap) и попытки эксплуатации, оценить эффективность базовой конфигурации правил.

### Топология стенда
- Защищаемая VM (IDS): `10.0.2.5` (`ubuntu-m`), интерфейс `eth0`.
- VM злоумышленника: `10.0.2.19` (`ubuntu-n`).
- Подсеть: `10.0.2.0/24`.
- Среда: Yandex Cloud (Security Groups разрешают ICMP, TCP:22, весь трафик внутри подсети).

### Методика тестирования

Были выполнены следующие тесты:
1. Сканирование портов Nmap: `-sA`, `-sT`, `-sS`, `-sV` по IP `10.0.2.5`.
2. Попытка генерации сигнатурного события: HTTP‑запрос с полезной нагрузкой `/etc/passwd` на порт 80.
3. Проверка логов Suricata: `fast.log`, `eve.json`.
4. Проверка статуса службы и конфигурации интерфейса.

### Результаты тестов

#### 1. Сканирование Nmap

| Тип сканирования | Команда | Статус порта 22 | Статус других портов | Наблюдение |
| --- | --- | --- | --- | --- |
| ACK scan | `nmap -sA 10.0.2.5` | unfiltered | unfiltered | Фаервол не блокирует ACK‑пакеты; хост виден |
| TCP connect | `nmap -sT 10.0.2.5` | open | closed | Подтверждено наличие SSH; остальные порты закрыты (RST) |
| SYN scan | `nmap -sS 10.0.2.5` | open | closed | Аналогично `-sT`; в облаке возможны потери SYN‑пакетов |
| Service detection | `nmap -sV 10.0.2.5` | open (OpenSSH 9.6p1) | closed | Определена версия SSH — потенциальная поверхность атаки |

#### 2. Тест на генерацию алерта (веб‑запрос)

Команда с VM злоумышленника:
```bash
curl -v http://10.0.2.5/?test=/etc/passwd

* connect to 10.0.2.5 port 80 ... failed: Connection refused
```
